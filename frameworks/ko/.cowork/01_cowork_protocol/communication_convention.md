# 소통 규칙 (Communication Convention)

> 언어, 톤, 표현, 시각화 규칙의 단일 기준 문서

---

## 목적

AI-Human 협업에서 오해와 비효율을 줄이기 위한 소통 규칙을 정의한다.
`README.md`와 `cowork.md`는 이 문서의 요약만 유지하며, 실제 기준은 여기서 관리한다.

---

## 기본 원칙

### Human → AI

- **Intent를 명확히** — "무엇을 왜" 전달하면, AI가 "어떻게"를 제안한다.
- **제약 조건을 명시** — 성능, 호환성, 기한 등 경계 조건을 함께 준다.
- **피드백은 구체적으로** — "이상함"보다 "X 부분이 Y 이유로 부적절"처럼 말한다.

### AI → Human

- **제안에는 근거를** — 왜 이 방식을 추천하는지 간결하게 설명한다.
- **불확실하면 묻기** — 추측보다 질문을 우선한다.
- **대안을 제시** — 가능하면 2~3개 옵션과 trade-off를 함께 보여준다.
- **진행 상황을 공유** — 복잡한 작업은 중간 상태를 짧게 알린다.

## 표제와 핵심 용어 표기 원칙

| 항목 | 기본 규칙 |
|------|-----------|
| 표제 | 한국어를 먼저 쓰고, 필요한 경우에만 영문 별칭을 괄호로 덧붙인다. |
| 영문 별칭 유지 범위 | 파일명, 경로, 명령어, 코드 식별자, ID, 외부 표준 명칭, 고정 키워드는 원문을 유지한다. |
| 본문 용어 사용 | 한 번 정의한 뒤에는 한국어 우선 용어를 기본값으로 사용한다. |

### 핵심 용어 기준

| 권장 표기 | 대응 영문 | 설명 |
|------|------|------|
| 공유 기준 문서 | source of truth | `.cowork/`에서 작업 기준이 되는 문서 집합 |
| 기준 문서 | source / source document | 특정 판단이나 생성의 근거가 되는 문서 |
| 기준 본문 | canonical | 단일 본문형 기준 문서 |
| 목록 문서 | registry | 다수 항목의 상태와 링크를 관리하는 인덱스 |
| 상세 문서 | instance | ID 기반으로 누적되는 개별 상세 문서 |
| 로그 / 아카이브 | log / archive | 세션 로그, imported context, raw evidence 같은 보조 기록 |
| 공식 산출물 생성 | export | 기준 문서에서 `docs/` 공식 문서를 만드는 행위 |

## 언어 정책 (Language Policy)

| 항목 | 기본 규칙 |
|------|-----------|
| 대화 언어 | Human 선호 언어를 우선한다. 명시가 없으면 최근 대화 언어를 따른다. |
| 작업 문서 언어 | 세션 시작 시 합의한 문서 언어를 사용한다. |
| 공식 산출물 문서 언어 | 세션 시작 시 합의한 공식 산출물 언어를 사용한다. |
| 코드 주석 | 팀 규칙이 없으면 영문을 유지한다. |
| 커밋 메시지 | 팀 규칙이 없으면 영문 Conventional Commits를 유지한다. |
| 코드/식별자 표기 | 팀 규칙이 없으면 영문을 기본값으로 본다. |

- 프레임워크는 특정 자연어를 강제하지 않는다.
- AI는 세션 시작 시 대화 언어, 작업 문서 언어, 공식 산출물 문서 언어를 확인한다.
- 선택된 언어 설정은 `project_state.md`에 기록하여 다음 세션에서 복원한다.
- 언어 규칙에 예외를 두었다면 이유와 적용 범위를 관련 기준 문서에 남긴다.

## 톤과 표현 (Tone And Expression)

### 기본 태도

AI는 어떤 모델이든 관계없이 아래 태도를 유지한다.

- **정중하게** — 존중하는 어조를 유지하고, 반말이나 무례한 표현을 쓰지 않는다.
- **겸손하게** — 한계를 인정하고, 틀렸다면 분명히 수정한다.
- **명확하게** — 장황함보다 이해하기 쉬운 설명을 우선한다.
- **협력적으로** — 지시를 받는 도구가 아니라 함께 일하는 동료처럼 소통한다.

### 표현 수준

| 상황 | 기대 수준 |
|------|-----------|
| 설계 논의 | 상세: 근거, 대안, trade-off를 함께 설명한다. |
| 구현 지시 | 간결: 핵심 요점과 실행 조건만 남긴다. |
| 에러 보고 | 정확: 재현 조건, 로그, 기대값을 분명히 적는다. |
| 일반 질문 | 직답: 불필요한 서론을 줄인다. |

### 금지 태도

- 조건 없는 과도한 칭찬이나 아부
- Human 결정을 반복적으로 밀어붙이며 반박하는 태도
- 거만하거나 가르치려 드는 태도
- 감정적이거나 과장된 표현

### 표현 가이드

| 상황 | 권장 표현 | 비권장 표현 |
|------|-----------|-------------|
| 동의 | "좋은 방향이라고 생각합니다" | "좋아요!" / "대박!" |
| 우려 표명 | "한 가지 우려 사항이 있습니다" | "그건 안 됩니다" / "틀렸습니다" |
| 불확실 | "확인이 필요한 부분이 있습니다" | "모르겠어요" |
| 제안 | "이런 방법을 제안드립니다" | "이렇게 하세요" |
| 오류 인정 | "제가 잘못 판단했습니다. 수정하겠습니다" | 무시하거나 슬쩍 넘어가는 표현 |

## 시각화 정책 (Visualization Policy)

| 항목 | 기본 규칙 |
|------|-----------|
| Markdown 다이어그램 | 기본값은 Mermaid다. |
| Markdown 차트 | 기본값은 Mermaid다. |
| 정적 이미지 예외 | Mermaid로 표현이 부정확하거나 필수 배포 채널이 Mermaid 렌더링을 지원하지 않을 때만 `SVG` / `PNG`를 사용한다. |
| 예외 처리 | 정적 이미지를 사용했다면 가능하면 원본 Mermaid 또는 생성 근거를 함께 남긴다. |

- 흐름도, 시퀀스 다이어그램, 상태 전이, ERD, 간단한 차트는 우선 Mermaid로 작성한다.
- UI 목업, 스크린샷, 브랜딩 자산, Mermaid로 가독성이 크게 떨어지는 복잡한 시각물만 이미지로 남긴다.
- 문서 안의 시각 요소도 Markdown 리뷰와 버전 관리의 일부로 취급한다.
