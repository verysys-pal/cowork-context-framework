# Team Board

> 팀 전체 역할·Task 할당 현황 — 누가 무엇을 하고 있는지, 의존 관계는 무엇인지

---

## 목적

프로젝트의 모든 Task가 누구(또는 어떤 역할)에게 배정되었는지,
현재 진행 상태와 의존 관계가 어떤지를 팀 전체가 공유하기 위한 보드다.

- `project_state.md`는 프로젝트 수준의 상태 요약이고
- 이 문서는 **팀원 수준의 할당과 진행**을 추적한다

---

## 팀 구성 모드

- [ ] **확정팀**: 실제 팀원이 존재 → 직접 역할 배정
- [ ] **사전배분**: 팀 미확정 → AI가 역할 슬롯 생성 후 Task 할당 → 이후 실제 팀원 매칭

> 확정팀이더라도 AI가 업무량 분석 결과 추가 인력이 필요하다고 판단하면,
> 가상 역할 슬롯을 추가 제안할 수 있다. Human 승인 후 등록한다.

## 작업 분배 가능 시점 (Task Distribution Readiness)

팀 프로젝트에서 작업을 여러 사람(또는 역할)에게 나눌 수 있는 시점은
**아키텍처와 모듈 경계가 확정된 이후**다.

### 전제 조건

| 조건 | 근거 | 확인 방법 |
|------|------|----------|
| Intent 승인 완료 | 프로젝트 목표와 범위가 합의됨 | `INT-NNN` 상태: Approved |
| 아키텍처 ADR 승인 | 모듈 경계, 계층 구조, 통신 방식이 확정됨 | 관련 ADR 상태: Approved |
| 모듈/기능 분해 완료 | 독립적으로 할당 가능한 작업 단위가 식별됨 | `task_registry.md`에 Task 목록 존재 |
| 인터페이스 계약 정의 | 모듈 간 의존 관계와 인터페이스가 명확함 | `interface_contract.md` 초안 이상 |

### Phase별 작업 분배 가능 범위

| Phase | 분배 가능 여부 | 설명 |
|-------|-------------|------|
| **Define** | 제한적 | 요구사항 수집은 영역별로 나눌 수 있으나, 전체 방향이 먼저 합의되어야 함 |
| **Design** | 부분 가능 | 아키텍처 확정 후 모듈별 상세 설계를 병렬로 진행 가능 |
| **Build** | **최적 시점** | 모듈 분해 + 인터페이스 계약이 있으면 독립적 구현이 가능 |
| **Verify** | 가능 | 기능별/모듈별 테스트를 병렬 수행 가능 |
| **Deliver** | 가능 | 산출물별로 담당자를 나눌 수 있음 |

### AI 자동 판단 기준

AI는 다음 조건이 **모두** 충족되면 "작업 분배 가능" 상태로 판단하고 Human에게 알린다:

1. 최소 1개 이상의 아키텍처 ADR이 Approved 상태
2. `task_registry.md`에 2개 이상의 독립 Task가 존재
3. 각 Task의 의존 관계가 명시되어 있음

> **1인 프로젝트**: 이 섹션은 해당하지 않으나, 추후 팀 확장 시 참조한다.

## 진행 규칙

- 담당자가 Task 상태를 변경하면 이 문서와 개인 `my_state.md`를 함께 갱신한다
- AI는 세션 시작 시 이 보드를 읽어 팀 전체 진행 상황을 파악한다
- 담당자 변경, Task 추가/분할은 Master 승인이 필요하다
- 역할 매칭 변경 시 해당 역할의 모든 Task 담당자가 자동 갱신된다
