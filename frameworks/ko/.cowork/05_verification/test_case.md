# Test Case

> 테스트 케이스 — 개별 검증 항목의 입력, 절차, 기대 결과를 정의한다

---

## 문서 정보

| 항목 | 내용 |
| --- | --- |
| 관련 Intent | 활성 Intent 또는 관련 `INT-*` 참조 |
| 관련 Milestone | 검증 대상 `MS-*` (해당 시) |
| 관련 Test Strategy | `test_strategy.md` |
| 관련 Verification Evidence | `verification_evidence.md` |
| 버전 | 프로젝트 기준 |

---

## 목적

테스트 전략(`test_strategy.md`)의 시나리오를 **실행 가능한 개별 케이스**로 분해하여,
누가 실행하더라도 동일한 결과를 재현할 수 있는 기준을 제공한다.

---

## 분할 승격 기준

- 기본값은 이 문서를 단일 canonical 문서로 유지하는 것이다. 아직 규모가 작으면 `test_cases/` 구조를 만들지 않는다.
- 아래 조건 중 하나를 만족하면 `test_cases/TC-*.md` 구조로의 **분할 승격 검토**를 시작한다.
  - 빈 예시를 제외한 실질 `TC-*` 케이스가 **12개를 넘길 때**
  - `최근 실행 결과 요약`의 실질 행이 **15개를 넘길 때**
  - **최근 3개 세션 중 2개 이상**이 같은 기능/같은 `MS-*`/같은 요구사항 묶음의 일부 `TC-*`만 반복 참조하고, 나머지 케이스는 계속 비참조일 때
- 분할이 승인되면 `test_case.md`는 전체 요약과 추적 인덱스를 유지하고, 상세 절차와 실행 이력은 `test_cases/TC-*.md`로 이동한다.
- 분할 후에도 `FR-*` / `NFR-*`, `TS-*`, `EV-*` 연결은 동일하게 유지해 `test_strategy.md`, `verification_evidence.md`, gate 판정 기준과의 추적성을 끊지 않는다.

---

## 작성 기준

- 이 문서에서는 HTML 주석형 placeholder 대신 `미정`, `미작성`, `없음`을 사용한다.
- 테스트 케이스가 없으면 최소 상태만 남기고, 실제 케이스가 생기면 `TC-*` 구조를 반복해 추가한다.
